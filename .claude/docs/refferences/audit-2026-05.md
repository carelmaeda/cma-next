1. The 60-second test
   First thing my eye lands on: the sticky top navbar with a "logo + green pulsing dot + 'Open for cool projects'" pill, then the hero line "Hi! My name is Carel." The pulsing "Open for cool projects" dot is louder than the hero — it's the only animated element competing with your name, and "cool projects" is a freelancer/junior signal in the navigation slot of a Design Lead site.
   Who Carel is, in 10 seconds: "Carel — leads product design at the intersection of UX strategy and front-end execution." Clear enough as a sentence, but the H1 is "Hi! My name is Carel." with the lead paragraph doing the actual positioning. The casual "Hi! My name is" frames the rest as a personal site, not a leader's portfolio. 7. Technical / SEO health snapshot
   Stack: Next.js 15 (App Router), React 19, TypeScript, Bootstrap 5.3 (loaded via CDN script tags in <head>), Framer Motion, react-icons, yet-another-react-lightbox, nodemailer + emailjs. Hosted (per metadataBase) at carelmaeda.com.
   Image optimization: Uses next/image consistently with explicit width/height and .webp source files. Good. Mixed loading="lazy" on the hero/profile images — the LCP image should not be lazy-loaded. The home page has no hero image so this is less critical, but the Image on /about profile is loading="lazy" and inside the above-fold About section.
   Font loading: Uses next/font/google for Inter and Urbanist — good. But applies both classNames to <html> (${inter.className} ${urbanist.className}), so Urbanist's class overrides Inter — you're loading two fonts and using one. Wasted byte.
   CSS framework: Bootstrap 5.3 JS is loaded from a CDN via <Script strategy="afterInteractive">, plus Popper from a different CDN. Both render-blocking-adjacent, both add third-party origins to the connection budget, both are needed only for the mobile hamburger. Heavy for one menu.
   Semantic HTML / a11y:
   Decorative-only H1 inside d-none (<div className="d-none"><h1>...</h1></div>) is a classic SEO-vs-design hack and an accessibility smell — screen readers may or may not read d-none (Bootstrap's d-none is display: none, so it won't). You have no visible H1 on the page. The semantic H1 is invisible to assistive tech and to humans.
   case-studies/birdseye and paygos-prototype repeat the same <h1 className="d-none">Case Studies</h1> pattern — same issue.
   Heading hierarchy on home is H2-led (Work, How I Work, About, Contact CTA) — fine, but no H1 means the page has no document title in the outline.
   Videos autoplay without captions or controls — accessibility issue for the videos on case study pages.
   Metadata: Root layout.tsx has good metadata with title template, description, OG, Twitter. Per-page metadata: none of the case study pages export metadata, so all four case studies inherit the default "Carel Maeda — Design Lead" title and home-page description. Each case study page is effectively invisible to search and to social sharing as a distinct document.
   Structured data: None. No JSON-LD Person or CreativeWork schema anywhere.
   Sitemap / robots: Neither src/app/sitemap.ts nor src/app/robots.ts exists. Case studies will not be in a sitemap.
   OG image: Custom OG image route exists at opengraph-image.tsx and renders nicely — this is a real strength and is being underused (no per-case-study OG variants).
   GA: Loaded inline in <head> via afterInteractive — fine. Hardcoded G-TGVPXPQKVJ is in the public repo, which is fine for GA4 measurement IDs.
   Build hygiene: package.json says "name": "next-proj" — default scaffold name still on a production portfolio.
   Verifiable craft check: The public repo URL in the footer means anyone can browse the same code, find the broken /work/*, /about, /resume, /archive routes, see the misspellings in the case studies, and read the commented-out skill carousel still sitting in Hero.tsx. A hiring manager who clicks "Developed with Next.js" is one click from your dirty laundry.
